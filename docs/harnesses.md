# Coding-agent harnesses — state, gates & phased workflows

Survey of popular GitHub harnesses/frameworks for **coding agents**, filtered for explicit
**workflow structure**: phases, state machines, spec-driven dev, verification gates, loop control.
Goal: find ideas to borrow into Jason's dotfiles skills (already has: superpowers-style phased skills,
a `total-tdd` CSV state machine, `goal-spec` verifiability gates, a binary linting rubric).

Star counts are approximate (mid-2026, from public listings/search; GitHub API was rate-limited in-sandbox).
Sorted by relevance to a **state + gates** focus.

| harness | stars | state/gate mechanism | what to borrow |
|---|---|---|---|
| [obra/superpowers](https://github.com/obra/superpowers) | ~170k | **Phase skills as mandatory workflow** — brainstorm → worktree → written plan → subagent TDD → fresh-agent code review → finish. Each phase is its own skill; next phase doesn't start until the prior one completes. State lives in the plan file + git worktree, not context. | The "fresh agent reviews each phase" gate; plan-file-as-state. Already Jason's base model — mine its newer skills for gate wording. |
| [eyaltoledano/claude-task-master](https://github.com/eyaltoledano/claude-task-master) | ~21k | **Explicit task state machine**: `pending / in-progress / done / review / deferred / cancelled` in `.taskmaster/state.json` + per-task `tasks.json`. `next` command gates work: a task is eligible only if status is pending/in-progress **AND all dependencies satisfied**; validates unique IDs, valid statuses, dep refs, **circular-dependency prevention**. Subtasks can depend on siblings or parent. | The dependency-DAG gate ("can't start until prereqs done") and circular-dep validation — a stronger version of `total-tdd`'s flat CSV. The `review` status as a distinct gated state. |
| [github/spec-kit](https://github.com/github/spec-kit) | ~40k | **Spec-driven pipeline**: `/specify → /plan → /tasks → /implement`. Each phase writes a markdown artifact the next phase reads ("nothing in the agent's head alone"). `/tasks` emits a dependency-ordered, `[P]`-parallel-marked, file-path-pinned task list with TDD ordering (tests before impl). | Artifact-per-phase handoff + the structured `/tasks` format (explicit file paths, `[P]` parallel markers, test-first ordering). Directly complements `goal-spec`. |
| [bmad-code-org/BMAD-METHOD](https://github.com/bmad-code-org/BMAD-METHOD) | ~49k | **4-phase pipeline** (Analysis→Planning→Architecture→Stories) driven by role-agents (PM/Architect/SM/Dev/QA). Explicit gates: `check-implementation-readiness` returns **PASS / CONCERNS / FAIL** before Phase 4; `code-review` gate before story completion; `sprint-status` tracks story state. Progressive doc context chains phases. | The **PASS/CONCERNS/FAIL** readiness gate as a discrete pre-implementation check (richer than binary) and role-agent handoffs. Fold a readiness-gate skill before `/goal` dispatch. |
| [humanlayer/12-factor-agents](https://github.com/humanlayer/12-factor-agents) | ~Hot/ref | **Principles, not a runtime.** Factor 5 "unify execution state," Factor 8 "own your control flow," Factor 4 "tools are structured outputs." Argues most real agents are deterministic code with LLM steps at strategic points, with custom control structures that can break the loop for human/long-running tasks. | The mental model: own the loop, make state explicit and serializable, treat tool calls as structured break-points. Design reference for any new harness/state machine, not a dep. |
| [langchain-ai/langgraph](https://github.com/langchain-ai/langgraph) | ~126k | **Literal state machine library**: app = directed graph; nodes = steps, edges (incl. conditional) = transitions; one typed `State` object flows through, updated via declared reducers. **Durable execution** (resume after failure) + human-in-the-loop (inspect/modify state mid-run). | The statechart vocabulary (typed state + reducers + conditional edges + checkpointing). If `total-tdd`'s CSV ever outgrows flat rows, this is the target shape. Library, not a coding harness per se. |
| [ruvnet/claude-flow](https://github.com/ruvnet/claude-flow) (now `ruflo`) | ~31–40k | **Swarm meta-harness**: hive-mind (queen/worker roles), SQLite memory at `.swarm/memory.db`, **SPARC methodology** as the workflow the swarm follows, stream-json agent-to-agent chaining, dependency/parallel orchestration. | SPARC as a named phase sequence; durable SQLite memory as cross-session state (vs. plan files). Heavy/complex — borrow patterns, don't adopt wholesale. |
| [All-Hands-AI/OpenHands](https://github.com/OpenHands/OpenHands) | ~78k | **Event-sourced architecture**: agent = function from event-history → next event, run in a loop; append-only `EventLog` of Actions/Observations is both memory and integration point. Auxiliary services (memory compression, **stuck detection**, security review, sub-agent delegation) hang off the event stream. | Append-only event log as the single source of truth (reproducible, resumable); **stuck-detection** as an auto-gate to break runaway loops. Pattern worth stealing for Ralph-style loops. |
| [SWE-agent/SWE-agent](https://github.com/SWE-agent/SWE-agent) | ~17k | **Agent-Computer Interface (ACI)**: tools redesigned to be LLM-friendly (structured output, guardrails, linter-on-edit) rather than explicit phases. State is the trajectory; "gates" are tool-level validation (e.g. edits rejected if they break syntax). `mini-swe-agent` is a ~100-line loop scoring >74% SWE-bench. | The ACI idea: gate *at the tool boundary* (reject bad edits, structured tool output) so the agent can't advance with broken state. Complements the binary linting rubric. |
| [Ralph Wiggum technique](https://github.com/ghuntley/how-to-ralph-wiggum) | ~ref (Huntley) | **Minimal loop, no engine**: `while :; do cat PROMPT.md \| claude -p; done`. State lives entirely in files + git history; the plan file on disk is shared state; fresh context every iteration. No internal gates — relies on the prompt to self-check and stop. | The "fresh context each loop, state on disk" discipline (already echoed in superpowers). Pair with OpenHands-style stuck-detection or a binary rubric to add the missing stop-gate. Jason already has the `ralph-wiggum` plugin/skill. |
| [buildermethods/agent-os](https://github.com/buildermethods/agent-os) | ~5k | **Standards + spec shaping** (v3): extracts codebase conventions into documented standards, injects the relevant ones per task, then shapes specs before building. Workflow is spec→standards→build; lighter on hard gates than spec-kit/BMAD. | "Standards as a system" — auto-inject repo conventions into the spec phase. A context-loading pattern more than a state machine; useful for the spec/lint side. |

Honorable mention: **[Amp](https://ampcode.com/manual)** (Sourcegraph, closed-source) — not a state-machine harness, but its **Oracle** (separate reasoning context invoked as a gate) and **subagent context multiplication** (garbage tokens stay in the subagent, only clean results return) are gate/isolation patterns worth noting; superpowers' fresh-review-agent is the same idea.

---

## Top 3 to study

1. **claude-task-master** — the cleanest off-the-shelf **dependency-DAG state machine** with a real `review` gate and circular-dep validation; a direct upgrade path for `total-tdd`'s flat CSV.
2. **github/spec-kit** — the tightest **artifact-per-phase, gated spec→plan→tasks→implement** pipeline, with a `/tasks` format (file paths + `[P]` parallel + test-first) that slots straight into `goal-spec`.
3. **BMAD-METHOD** — for its explicit **PASS/CONCERNS/FAIL readiness gate** and code-review gate between phases — a richer, non-binary gate model to add before `/goal` dispatch.

## Patterns worth adopting

1. **Dependency-DAG gating, not flat rows.** Adopt Taskmaster's rule: a step is eligible only when `status ∈ {pending,in-progress}` AND all deps satisfied, with circular-dep validation. Evolve `total-tdd`'s CSV from a flat list into a small DAG so the "what's next" decision is mechanical, not judgment.
2. **A distinct `review` / readiness gate state.** Both Taskmaster (`review` status) and BMAD (`PASS/CONCERNS/FAIL` readiness check) make "done" a *two-step* gate: work-complete → independently-verified. Add a readiness-gate skill that runs the binary linting rubric (and/or adversarial-review) before a task can flip to `done` or before `/goal` dispatches.
3. **Append-only event log + stuck-detection for loops.** From OpenHands: keep a durable, append-only log of actions/observations as the source of truth, and bolt on automatic stuck-detection to break runaway loops. This is the missing stop-gate for Ralph-style/`ralph-wiggum` runs.
4. **Gate at the tool boundary (ACI).** From SWE-agent: reject state-breaking edits at the tool level (lint/syntax-check on edit) so the agent literally can't advance with broken state — a deterministic complement to the prose-level phase gates Jason already has.
